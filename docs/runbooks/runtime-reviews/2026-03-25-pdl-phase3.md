Date: 2026-03-25
Owner: Pedro / Keel runtime review loop
Priority: Phase 3 callable-provider verification

## Why PDL

From the callable inventory (`GET /v1/proxy/services`), PDL remained the clearest Phase 3 risk:
- proxy inventory exposes the provider as `pdl`
- capability resolution exposes the same provider as `people-data-labs`
- working memory already flagged the PDL execution path as broken
- it had no explicit Runtime-verified review entries despite callable status

## Likely cause

The strongest current hypothesis is a **slug / provider normalization mismatch** across layers:
- proxy registry uses `pdl`
- managed credential injection is keyed on `people-data-labs`
- service catalog and capability resolution also use `people-data-labs`
- older capability-service rows still reference `pdl`

That inconsistency is the cleanest explanation for why the provider is publicly callable in one layer while runtime execution remains broken in another.

## Phase 3 verdict

**PDL is not Phase-3-verified.**

Direct provider runtime is healthy, but Rhumb Resolve execution is still broken. This provider should remain on the callable verification queue until the Resolve path returns a successful structured response and emits a normal execution log row.

## 11:53 PM rerun — managed-config alias bug isolated

A fresh production rerun after the billing-row fix changed the failure shape enough to isolate a second root cause.

### Production behavior
- Estimate with `provider=people-data-labs&credential_mode=rhumb_managed` returned **200**
- Explicit `rhumb_managed` execute still returned **503 Service Unavailable**
- `auto` returned **400** demanding BYO `method` + `path`
- Explicit `byo` returned **200 OK**
- Direct PDL control remained **200 OK**

### Interpretation
That pattern means the upstream provider is healthy and Rhumb BYO proxying is healthy. The remaining defect lives specifically in the **Rhumb-managed config resolution path**.

### New root cause
`rhumb_managed_capabilities` stores the provider as canonical `people-data-labs`, but managed-config lookup normalized requested providers to proxy alias `pdl` before querying the table.

Consequences:
1. explicit `rhumb_managed` could not find a managed config → 503
2. `auto` also failed to find a managed config and silently degraded to BYO mode, which then rejected the request because BYO requires explicit `method` + `path`

### Repo fix implemented
- `packages/api/services/rhumb_managed.py`
  - added alias-candidate lookup so managed config resolution tries canonical and proxy forms
- `packages/api/routes/capability_execute.py`
  - stopped prematurely normalizing the requested provider before managed-config lookup
- regression coverage added in `packages/api/tests/test_rhumb_managed.py`

### Validation after fix
- targeted regression tests for managed-config alias lookup pass
- local managed executor proof with the real PDL upstream key now succeeds for `service_slug='pdl'` and returns **200 upstream**

### Current verdict
PDL remains **not yet reconfirmed live in production** until the new managed-config alias fix deploys, but the remaining bug is no longer mysterious: it is isolated, fixed in repo, and locally validated against the real PDL upstream.
