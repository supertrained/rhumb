# Phase 3 runtime review — People Data Labs (PDL)

Date: 2026-03-25
Owner: Pedro / Keel runtime review loop
Priority: Phase 3 callable-provider verification

## Why PDL

From the callable inventory (`GET /v1/proxy/services`), PDL remained the clearest Phase 3 risk:
- proxy inventory exposes the provider as `pdl`
- capability resolution exposes the same provider as `people-data-labs`
- working memory already flagged the PDL execution path as broken
- it had no explicit Runtime-verified review entries despite callable status

## Test setup

### Rhumb Resolve execution
- Endpoint: `POST /v1/capabilities/data.enrich_person/execute`
- Provider: `people-data-labs`
- Modes tested: `rhumb_managed`, `auto`, and `byo`
- Input: public LinkedIn profile for Satya Nadella
  - `profile=https://www.linkedin.com/in/satyanadella/`

### Direct provider control
- Endpoint: `GET https://api.peopledatalabs.com/v5/person/enrich`
- Auth: direct PDL API key
- Input: same LinkedIn profile

## Results

### Rhumb Resolve
- Result: **failed**
- HTTP status: **503**
- Body: plain-text `Service Unavailable`
- Observed behavior:
  - failure reproduced consistently across `rhumb_managed`, `auto`, and `byo`
  - no matching `capability_executions` row was written for the test agent
  - failure appears to occur before normal execution logging completes

### Direct PDL control
- Result: **succeeded**
- HTTP status: **200**
- Response highlights:
  - `status: 200`
  - `likelihood: 9`
  - returned a structured person record for the supplied LinkedIn profile

## Comparison

| Dimension | Rhumb Resolve | Direct PDL |
|---|---|---|
| Reachability | Fails with 503 | Healthy |

| Logging | No execution row observed | Normal provider response |
| Operator conclusion | Broken integration path | Provider API healthy |

## Likely cause

The strongest current hypothesis is a **slug / provider normalization mismatch** across layers:
- proxy registry uses `pdl`
- managed credential injection is keyed on `people-data-labs`
- service catalog and capability resolution also use `people-data-labs`
- older capability-service rows still reference `pdl`

That inconsistency is the cleanest explanation for why the provider is publicly callable in one layer while runtime execution remains broken in another.

## Phase 3 verdict

**PDL is not Phase-3-verified.**

Direct provider runtime is healthy, but Rhumb Resolve execution is still broken. This provider should remain on the callable verification queue until the Resolve path returns a successful structured response and emits a normal execution log row.

## Follow-up

1. Normalize provider slug handling (`pdl` vs `people-data-labs`) across proxy, capability resolution, and managed execution.
2. Re-run `data.enrich_person` with the same public-profile payload.
3. Once Rhumb succeeds, publish/update runtime-backed review coverage for the service.
