# Firebase Auth — Agent-Native Service Guide

> **AN Score:** 6.35 · **Tier:** L3 · **Category:** Authentication & Identity

---

## 1. Synopsis
Firebase Auth is a managed identity service that allows agents to programmatically manage user lifecycles, verify identities, and secure cross-service communication. For agents, it serves as a critical security layer, enabling them to verify the identity of the human users they represent or to manage access for other sub-agents. It abstracts complex OAuth2, SAML, and OIDC flows into a unified API. The service is highly attractive for agent deployments due to its generous free tier (50,000 monthly active users for most providers) and deep integration with Google Cloud’s IAM. Agents can execute administrative tasks like password resets, account linking, and token verification without human intervention, making it a "Ready" (L3) service for autonomous identity management.

---

## 2. Connection Methods

### REST API
The Firebase Auth REST API (part of the Google Identity Toolkit) is available at `identitytoolkit.googleapis.com`. While functional for basic operations like signing in with a password or exchanging refresh tokens, it is generally lower-level than agents require. Most autonomous workflows should favor the Admin SDK for broader project-level permissions.

### SDKs
The **Firebase Admin SDK** is the primary interface for agents. Available in Python (`firebase-admin`), Node.js (`firebase-admin`), Go, and Java, it provides elevated privileges to bypass client-side restrictions. Agents use the Admin SDK to perform CRUD operations on user records, mint custom tokens, and verify ID tokens generated by client applications. This is the preferred method for agent-to-service integration.

### Webhooks
Firebase Auth does not support native outbound webhooks directly. Instead, it uses **Blocking Functions** (via Cloud Functions for Firebase). This allows an agent to intercept registration or sign-in events in real-time to perform validation, deny access, or enrich user metadata before the transaction completes.

### Auth Flows
Agents typically interact with Firebase Auth via **Service Account Keys** (JSON). For agent-to-user interactions, the agent verifies a user's `idToken` sent in a request header. For agent-to-agent interactions, the **Custom Token** flow allows an agent to act as an identity provider, minting a token that another agent can use to authenticate against a Firebase project.

## 3. Key Primitives

| Primitive | Endpoint/Method | Description |
| :--- | :--- | :--- |
| **Create User** | `auth.create_user()` | Programmatically creates a new user record with specific UID, email, or properties. |
| **Verify ID Token** | `auth.verify_id_token()` | Validates a JWT sent by a client; ensures the user is authenticated and the token is not expired. |
| **Update User** | `auth.update_user()` | Modifies existing user attributes (e.g., disabling an account, changing display names). |
| **List Users** | `auth.list_users()` | Batch retrieves user records; supports pagination for large-scale directory audits. |
| **Delete User** | `auth.delete_user()` | Permanently removes a user record and revokes their existing refresh tokens. |
| **Custom Claims** | `auth.set_custom_user_claims()` | Attaches granular permissions (roles) to a user's JWT for fine-grained access control. |
| **Mint Custom Token** | `auth.create_custom_token()` | Generates a short-lived token for a specific UID to facilitate secure cross-agent handoffs. |

## 4. Setup Guide

### For Humans
1. Go to the [Firebase Console](https://console.firebase.google.com/) and create or select a project.
2. Navigate to **Authentication** and click "Get Started."
3. Enable your desired Sign-in providers (e.g., Email/Password, Google).
4. Go to **Project Settings** > **Service Accounts**.
5. Click **Generate New Private Key** to download the JSON credentials file.
6. Store this JSON securely; it grants full administrative access to your user database.

### For Agents
1. **Environment Injection**: Ensure the service account JSON path is available via `GOOGLE_APPLICATION_CREDENTIALS`.
2. **Library Installation**: Install the admin library (e.g., `pip install firebase-admin`).
3. **Initialization**: Use the default credential provider to initialize the app.
4. **Validation**: Execute a "List Users" call with a limit of 1 to verify connectivity.
```python
import firebase_admin
from firebase_admin import auth, credentials

# Connection Validation
if not firebase_admin._apps:
    cred = credentials.ApplicationDefault()
    firebase_admin.initialize_app(cred)
    
# Test connectivity
page = auth.list_users(max_results=1)
print(f"Connection Verified: Found {len(page.users)} users.")
```

## 5. Integration Example

```python
import firebase_admin
from firebase_admin import auth, exceptions

# Initialize the Admin SDK
# Assumes GOOGLE_APPLICATION_CREDENTIALS env var is set
default_app = firebase_admin.initialize_app()

def provision_agent_user(email, display_name):
    try:
        # 1. Check if user exists, or create new
        try:
            user = auth.get_user_by_email(email)
            print(f"User already exists: {user.uid}")
        except auth.UserNotFoundError:
            user = auth.create_user(
                email=email,
                display_name=display_name,
                email_verified=True
            )
            print(f"Created new user: {user.uid}")

        # 2. Set custom claims for agent permissions
        # This allows the agent to verify the user's role in future requests
        auth.set_custom_user_claims(user.uid, {'role': 'automated_agent', 'clearance': 'L2'})
        
        # 3. Generate a password reset link for the human owner
        link = auth.generate_password_reset_link(email)
        return {"uid": user.uid, "setup_link": link}

    except exceptions.FirebaseError as e:
        # Structured error handling for agent decision making
        print(f"ErrorCode: {e.code} - Message: {str(e)}")
        return None

# Execute provisioning
result = provision_agent_user("agent.001@example.com", "Storage Bot")
```

## 7. Agent-Native Notes

*   **Idempotency**: The `create_user` method is not strictly idempotent if only email is provided. To ensure idempotency, agents should generate and provide a deterministic `uid` based on the user's external ID.
*   **Retry Behavior**: Agents should implement exponential backoff for `429` (Quota Exceeded) and `503` (Service Unavailable). Most SDKs handle internal retries for socket timeouts but not for rate limits.
*   **Error Codes**: Firebase returns string-based error codes (e.g., `auth/email-already-exists`). Agents should use these for branching logic rather than parsing error messages.
*   **Schema Stability**: Extremely high. The Firebase Auth schema for user objects has remained consistent for years, making it safe for long-term autonomous deployments.
*   **Cost-per-operation**: Effectively $0 for the first 50k MAU. For agents, this makes "Identity-per-Task" patterns economically viable.
*   **Token Verification**: Agents should perform `verify_id_token(id_token, check_revoked=True)` to ensure they aren't acting on behalf of a compromised or deleted session.
*   **Token Expiry**: ID tokens expire after 1 hour. Agents must be programmed to handle `auth/id-token-expired` by requesting a refresh from the client or re-authenticating.

## 8. Rhumb Context: Why Firebase Auth Scores 6.35 (L3)

Firebase Auth’s **6.35 score** reflects a robust, production-ready identity layer that is slightly hampered by the friction of Google Cloud's credential management:

1. **Execution Autonomy (7.5)** — The Admin SDK is exceptionally powerful. Agents can perform nearly any identity task—from minting custom JWTs to modifying security claims—without human approval. The structured nature of the `UserRecord` object allows agents to reliably parse and manipulate user data.

2. **Access Readiness (4.8)** — This is the primary drag on the score. Bootstrapping an agent requires navigating the Google Cloud Console to generate Service Account JSON keys. This process is difficult to automate fully without existing high-level IAM permissions, creating a "chicken and egg" problem for completely autonomous agent self-provisioning.

3. **Agent Autonomy (7.0)** — The service excels in governance. Google Cloud Audit Logs provide a perfect paper trail of every action an agent takes within the Auth service. Integration with Cloud Functions (Blocking Triggers) allows agents to act as "gatekeepers" for the authentication process itself, enabling autonomous security policies.

**Bottom line:** Firebase Auth is the premier choice for agents operating within the Google ecosystem or those needing a high-scale, low-cost identity provider. Its L3 status indicates it is ready for critical production workflows, provided the initial service account handshake is managed.

**Competitor context:** **Auth0 (7.2)** scores higher on Access Readiness due to its more modern API-first Management Dashboard, but carries much higher costs. **Supabase Auth (6.8)** offers better integration for agents already using PostgreSQL but lacks the global scale and enterprise governance features of Google's identity backbone.
